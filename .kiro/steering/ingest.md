# /ingest — 새 원본을 세컨드 브레인 위키로 인제스트

01_raw/ 폴더에 새로 추가된 파일을 찾아 LLM Wiki 패턴에 따라 위키로 인제스트하는 워크플로우.

## 트리거

사용자가 "인제스트해줘" 또는 "/ingest"라고 요청할 때 실행.

## 전체 프로세스

### 1단계: 새 파일 탐지

1. `01_raw/` 하위 폴더를 탐색하여 새로 추가된 파일을 찾는다
2. `02_wiki/log.md`를 읽어 이미 인제스트된 원본 경로를 확인한다
3. log에 없는 새 파일만 인제스트 대상으로 선정한다
4. PDF 파일이 있으면 마크다운 변환 여부를 확인한다:
   - 같은 이름의 폴더 안에 `.md` 파일이 있으면 변환 완료
   - 없으면 사용자에게 `marker_single` 변환을 안내한다

### 1.5단계: 네이밍 정규화 (PDF 리네이밍)

marker 변환 후 제목이 식별되면 파일명을 정규화한다:

1. 변환된 마크다운에서 제목을 추출한다 (첫 번째 `#` 헤딩 또는 메타데이터)
2. 제목을 슬러그로 변환한다 (영어 소문자 + 하이픈, kebab-case, 최대 80자)
3. 현재 파일명이 슬러그와 다르면 리네이밍을 수행한다
4. 리네이밍은 01_raw/ 내에서만 수행한다

**네이밍 컨벤션:**
| 유형 | 형식 |
|------|------|
| 논문 | `{제목-슬러그}/{제목-슬러그}.md` |
| 기사/유튜브/메모/코드 | `{YYYY-MM-DD}_{제목-슬러그}.md` |

### 2단계: 원본 읽기 및 소스 요약 생성

1. 원본 파일을 읽는다 (01_raw/는 절대 수정하지 않는다)
2. 소스 요약을 작성한다:
   - **핵심 주장 (Key Claims)**: 원본의 핵심 메시지 3~5개
   - **언급된 엔티티**: 사람, 도구, 조직, 모델 등
   - **다루는 개념 (Concepts)**: 핵심 기술 개념들
3. 사용자에게 3가지 질문을 한다:
   - "이 글을 왜 캡처했어?"
   - "지금 하고 있는 일과 어떻게 연결돼?"
   - "이걸로 뭘 해보고 싶어?"
4. 사용자 답변을 받을 때까지 기다린다

### 3단계: 02_wiki/ 반영

사용자 답변을 받으면 아래를 수행한다:

#### 3-1. 소스 요약 페이지 생성
- 경로: `02_wiki/sources/{슬러그}.md`
- 필수 frontmatter: title, created, updated, tags, sources, type, status
- 필수 섹션: Kernel(핵심), 핵심 주장, 캡처 맥락, 관련 개념, 관련 엔티티

#### 3-2. 엔티티 페이지 생성/갱신
- 경로: `02_wiki/entities/{슬러그}.md`

#### 3-3. 개념 페이지 생성/갱신
- 경로: `02_wiki/concepts/{슬러그}.md`

#### 3-4. index.md 갱신
- `02_wiki/index.md`의 해당 카테고리에 새 페이지 항목 추가

#### 3-5. log.md 갱신
- `02_wiki/log.md`에 append-only로 기록 추가

#### 3-6. 그래프 재빌드
- `python3 .scripts/build_graph.py` 실행하여 00_graphify-out/ 갱신

## 규칙

- 01_raw/ 폴더의 파일은 **절대 수정하지 않는다** (읽기 전용)
- 모든 02_wiki 페이지에 YAML frontmatter를 포함한다
- 위키링크 `[[페이지명]]`으로 페이지 간 연결을 유지한다
- 핵심(Kernel)을 먼저 짚고, 세부사항은 그 다음에 서술한다
- 한국어 기본, 기술 용어는 영어 병기
- 한 번에 하나의 원본만 인제스트한다
