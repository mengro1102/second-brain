## 영상 내용

6.3.1에서 양자화의 기본 원리를 살펴보았습니다. 부동소수점 가중치를 더 낮은 비트의 정수로 변환하면 모델 크기가 줄어들고, 메모리 바운드 상황에서 추론 속도가 빨라진다는 것을 확인했습니다. 또한 학습이 끝난 모델에 양자화를 적용하는 Post-Training Quantization(PTQ) 방식이 대규모 언어 모델에서 현실적인 선택이라는 점도 다루었습니다. 그런데 PTQ를 실제로 적용하려 하면, 구체적으로 어떤 도구를 써야 하는지, 도구마다 어떤 차이가 있는지라는 질문에 부딪힙니다. 현재 LLM 양자화 생태계에서 가장 널리 쓰이는 세 가지 방식이 GPTQ, AWQ, GGUF입니다. 이 단원에서는 각 방식의 원리와 특성을 비교하고, 서빙 프레임워크에서의 지원 현황을 살펴본 뒤, 실무에서 어떤 기준으로 양자화 방식을 선택하는지 정리합니다.

**GPTQ** 부터 살펴보겠습니다. GPTQ는 "Accurate Post-Training Quantization for Generative Pre-trained Transformers"의 약자로, 대규모 언어 모델을 대상으로 설계된 PTQ 기법입니다. 6.3.1에서 설명한 단순 양자화는 가중치의 최솟값과 최대값으로 스케일 팩터를 구한 뒤 일괄 변환하는 방식이었습니다. 이 방식은 빠르지만, 모든 가중치를 동일한 기준으로 변환하기 때문에 중요한 가중치에서 큰 오차가 발생할 수 있습니다. GPTQ는 이 문제를 해결하기 위해, 가중치를 한 번에 변환하지 않고 열(column) 단위로 순차 처리하면서, 앞선 열에서 발생한 양자화 오류를 뒤의 열에서 보상하는 전략을 사용합니다.

GPTQ의 동작 원리를 단계별로 설명하겠습니다. GPTQ는 모델의 각 층(layer)에 있는 가중치 행렬을 독립적으로 양자화합니다. 하나의 가중치 행렬을 양자화할 때, 소량의 보정 데이터(calibration data)를 모델에 통과시켜 각 층의 입력 활성화(activation) 값을 수집합니다. 이 활성화 값은 가중치가 실제 추론에서 어떻게 사용되는지를 반영합니다. 그다음, 가중치 행렬의 열을 하나씩 양자화합니다. 한 열을 양자화하면 반올림 오차가 생기는데, GPTQ는 이 오차가 해당 층의 전체 출력에 미치는 영향을 계산합니다. 그리고 아직 양자화하지 않은 나머지 열의 값을 미세하게 조정하여, 앞서 발생한 오차를 상쇄합니다. 이 과정을 마지막 열까지 반복하면, 층 전체의 양자화 오류가 최소화됩니다.

```
GPTQ 양자화 과정 (한 가중치 행렬 기준)

1. 보정 데이터를 모델에 통과시켜 입력 활성화 수집
2. 가중치 행렬의 열을 순서대로 처리:

   열 1    열 2    열 3    ...   열 N
   [양자화] [대기]  [대기]        [대기]
      |
      +-- 오차 발생
      |
      +-- 오차를 열 2~N에 분배하여 보상
              |
              [양자화] [대기]  ...   [대기]
                 |
                 +-- 오차를 열 3~N에 분배
                         ...
                                    [양자화]

3. 모든 열 처리 완료 -> 층 전체 오류 최소화
```

GPTQ는 이 보상 메커니즘 덕분에 4비트 양자화에서도 원본 모델과 유사한 품질을 유지합니다. 보정 데이터는 보통 128개 정도의 텍스트 샘플이면 충분하며, 70억 파라미터 모델을 단일 GPU에서 수 분 내에 양자화할 수 있습니다. GPTQ로 양자화된 모델은 GPU에서 실행하도록 설계되어 있습니다. 양자화된 가중치는 추론 시 GPU 커널에서 역양자화되어 부동소수점 연산에 사용됩니다.

GPTQ를 실제로 적용하려면 AutoGPTQ 같은 라이브러리를 사용합니다. 아래는 Hugging Face Transformers와 AutoGPTQ를 사용하여 모델을 양자화하는 예시입니다.

```python
from transformers import AutoModelForCausalLM, AutoTokenizer, GPTQConfig

model_id = "meta-llama/Llama-3.1-8B-Instruct"
tokenizer = AutoTokenizer.from_pretrained(model_id)

# 보정 데이터 준비
calibration_texts = ["양자화는 모델의 가중치 정밀도를 낮추는 기법입니다.", ...]

# GPTQ 양자화 설정
quantization_config = GPTQConfig(
    bits=4,
    dataset=calibration_texts,
    tokenizer=tokenizer
)

# 양자화 수행
model = AutoModelForCausalLM.from_pretrained(
    model_id,
    quantization_config=quantization_config,
    device_map="auto"
)
```

`bits=4` 는 가중치를 4비트 정수로 양자화하겠다는 설정입니다. `dataset` 에는 보정 데이터를 전달합니다. `device_map="auto"` 는 모델을 사용 가능한 GPU에 자동으로 배치하라는 의미입니다. 양자화가 완료되면 모델을 저장하여 이후 로드 시 다시 양자화할 필요 없이 바로 사용할 수 있습니다.

다음으로 **AWQ(Activation-aware Weight Quantization)** 를 살펴보겠습니다. AWQ는 GPTQ와 같은 PTQ 기법이지만, 접근 방식이 다릅니다. GPTQ가 열 단위로 오류를 보상하는 데 초점을 맞추었다면, AWQ는 "모든 가중치가 동등하게 중요하지 않다"는 관찰에서 출발합니다.

AWQ의 핵심 아이디어를 이해하려면, 활성화 값과 가중치의 관계를 먼저 알아야 합니다. 신경망에서 한 층의 출력은 입력 활성화와 가중치의 행렬 곱으로 계산됩니다. 이때 입력 활성화의 크기(magnitude)가 큰 채널에 연결된 가중치는, 최종 출력에 더 큰 영향을 미칩니다. 예를 들어, 입력 활성화가 \[0.01, 5.0, 0.03\]이고 이에 대응하는 가중치가 \[w1, w2, w3\]일 때, w2의 양자화 오류는 5.0이 곱해져 출력에 크게 반영되지만, w1이나 w3의 오류는 작은 활성화 값에 곱해져 출력에 거의 영향을 주지 않습니다.

AWQ는 이 관찰을 활용합니다. 보정 데이터를 통과시켜 각 채널의 평균 활성화 크기를 측정한 뒤, 활성화가 큰 채널에 연결된 가중치를 "중요한 가중치"로 식별합니다. 그런 다음 이 중요한 가중치의 양자화 오류를 줄이기 위해, 양자화 전에 가중치에 채널별 스케일링을 적용합니다. 중요한 가중치는 스케일을 키워 양자화 해상도를 높이고, 덜 중요한 가중치는 상대적으로 거칠게 양자화해도 전체 출력에 미치는 영향이 작으므로 무방합니다.

```rust
AWQ의 핵심 원리

입력 활성화:  [0.01, 5.00, 0.03]
               작음   큼    작음

대응 가중치:  [w1,   w2,   w3]
               |      |      |
               v      v      v
출력 기여도:  작음    큼    작음

-> w2가 중요한 가중치
-> w2의 양자화 해상도를 높이기 위해 스케일 조정

스케일 적용 전: w2 = 0.347  -> INT4로 양자화하면 오차 큼
스케일 적용 후: w2 x s = 0.347 x 4 = 1.388 -> INT4 해상도에서 오차 작음
(추론 시 역양자화 후 s로 나누어 원래 스케일 복원)
```

AWQ의 장점은 GPTQ보다 양자화 속도가 빠르다는 것입니다. GPTQ는 열 단위로 순차 처리하면서 매번 오류를 재계산해야 하지만, AWQ는 채널별 스케일을 한 번 결정하면 양자화를 일괄 수행할 수 있습니다. 또한 AWQ는 보정 데이터에 대한 의존도가 GPTQ보다 낮은 편입니다. 중요 가중치를 식별하는 기준이 활성화의 통계적 분포이므로, 소량의 보정 데이터로도 안정적인 결과를 얻습니다.

이제 **GGUF** 를 살펴보겠습니다. GGUF는 GPTQ나 AWQ와 성격이 다릅니다. GPTQ와 AWQ가 양자화 알고리즘이라면, GGUF는 양자화된 모델을 저장하는 **파일 포맷** 입니다. GGUF를 이해하려면 먼저 llama.cpp라는 프로젝트의 배경을 알아야 합니다.

llama.cpp는 C/C++로 작성된 LLM 추론 엔진입니다. 이 프로젝트가 등장한 배경은 다음과 같습니다. 대부분의 LLM 서빙 프레임워크(vLLM, TGI 등)는 NVIDIA GPU와 CUDA를 필요로 합니다. 하지만 모든 환경에 NVIDIA GPU가 있는 것은 아닙니다. 개인 노트북, Apple Silicon Mac, 라즈베리파이 같은 소형 장치에서도 LLM을 실행하고 싶은 수요가 있습니다. llama.cpp는 이 수요에 대응하여, GPU 없이 CPU만으로도 LLM을 실행할 수 있도록 설계되었습니다. 물론 GPU가 있으면 GPU 가속도 활용합니다.

llama.cpp가 CPU에서 효율적으로 추론을 수행하려면, 모델 가중치가 극도로 작아야 합니다. CPU의 메모리 대역폭은 GPU HBM에 비해 훨씬 좁기 때문입니다. 이를 위해 llama.cpp는 자체적인 양자화 방식과 파일 포맷을 개발했습니다. 초기에는 GGML이라는 포맷을 사용했으나, 이후 확장성과 호환성을 개선한 **GGUF(GGML Unified Format)** 로 대체되었습니다.

GGUF 포맷의 특징은 모델의 가중치, 토크나이저 정보, 모델 아키텍처 메타데이터를 하나의 파일에 담는다는 점입니다. 별도의 설정 파일이나 토크나이저 파일 없이 GGUF 파일 하나만 있으면 모델을 실행할 수 있습니다. 이 설계는 모델을 다운로드하고 공유하기 편리하게 만듭니다.

GGUF가 지원하는 양자화 수준은 매우 다양합니다. Q2\_K, Q3\_K\_S, Q3\_K\_M, Q4\_0, Q4\_K\_M, Q5\_K\_M, Q6\_K, Q8\_0 등 여러 양자화 변형이 있습니다. 이 이름에서 Q 다음의 숫자는 비트 수를 나타냅니다. K가 붙은 변형은 k-quant라는 기법을 사용하는데, 이는 가중치 블록마다 서로 다른 비트 수를 혼합하여 중요한 부분은 높은 비트로, 덜 중요한 부분은 낮은 비트로 양자화합니다. S, M, L은 각각 Small, Medium, Large를 의미하며, 높은 비트를 사용하는 비율이 다릅니다. 예를 들어 Q4\_K\_M은 4비트 기반이되 일부 블록에 더 높은 비트를 혼합하여 품질을 보완한 중간 수준의 변형입니다.

```
GGUF 양자화 변형 예시 (7B 모델 기준 대략적 크기)

변형       비트 수      대략적 크기    품질
------     --------     ----------     ---------
Q2_K       2비트 혼합   약 2.7 GB      낮음
Q3_K_M     3비트 혼합   약 3.3 GB      보통 이하
Q4_0       4비트 균일   약 3.8 GB      보통
Q4_K_M     4비트 혼합   약 4.1 GB      양호
Q5_K_M     5비트 혼합   약 4.8 GB      좋음
Q6_K       6비트 혼합   약 5.5 GB      매우 좋음
Q8_0       8비트 균일   약 7.2 GB      원본에 근접
```

이 표에서 볼 수 있듯이, GGUF는 2비트부터 8비트까지 넓은 범위의 양자화 수준을 제공합니다. 사용자는 자신의 하드웨어 사양과 원하는 품질 수준에 맞춰 적절한 변형을 선택할 수 있습니다. 이 유연성이 GGUF가 개인 사용자 커뮤니티에서 인기 있는 이유 중 하나입니다.

GGUF 모델을 실행하는 방법은 llama.cpp를 직접 사용하거나, llama.cpp 기반의 사용자 친화적인 도구를 사용하는 것입니다. llama.cpp 자체는 명령줄 도구이고, Ollama나 LM Studio 같은 프로젝트가 llama.cpp를 내장하여 더 쉬운 인터페이스를 제공합니다.

이제 세 가지 양자화 방식의 **품질과 속도를 비교** 하겠습니다. 품질은 원본 모델 대비 양자화 모델의 성능 저하 정도를 의미하며, 보통 펄플렉서티(perplexity) 같은 지표로 측정합니다. 펄플렉서티는 모델이 다음 토큰을 예측하는 확실성을 나타내는 수치로, 값이 낮을수록 좋습니다.

같은 4비트 양자화라 하더라도 GPTQ, AWQ, GGUF(Q4\_K\_M)의 품질은 조금씩 다릅니다. GPTQ와 AWQ는 모두 보정 데이터를 활용하여 양자화 오류를 줄이므로, 단순 양자화보다 품질이 높습니다. AWQ는 중요 가중치에 집중하는 전략 덕분에 동일 비트 수에서 GPTQ와 비슷하거나 약간 더 나은 품질을 보이는 경우가 많습니다. GGUF의 k-quant 변형도 비트 혼합 전략으로 단순 균일 양자화보다 품질이 좋지만, GPTQ나 AWQ에 비해서는 보정 데이터를 활용한 최적화가 덜 정교합니다.

속도 측면에서는 실행 환경에 따라 차이가 큽니다. GPU 환경에서는 GPTQ와 AWQ 모델이 최적화된 CUDA 커널을 활용하여 빠른 추론 속도를 보입니다. CPU 환경에서는 GGUF가 llama.cpp의 최적화된 CPU 커널 덕분에 압도적으로 유리합니다. GPTQ와 AWQ 모델은 CPU에서 실행하도록 설계되지 않았으므로 CPU 추론에는 적합하지 않습니다.

```
양자화 방식별 특성 비교 (4비트 기준)

항목              GPTQ              AWQ               GGUF (Q4_K_M)
-----------       ------            ------            --------
양자화 원리       열 단위 오류 보상  활성화 기반        k-quant 비트 혼합
                                    중요 가중치 보호
보정 데이터       필요 (128샘플)     필요 (소량)        불필요 (단순 통계)
양자화 속도       보통 (수 분~수십 분) 빠름 (수 분)     빠름
4비트 품질        높음              높음~매우 높음      양호
주 실행 환경      GPU               GPU               CPU + GPU
대표 도구         AutoGPTQ          AutoAWQ           llama.cpp
파일 포맷         safetensors       safetensors       GGUF 단일 파일
```

다음으로 **양자화 모델의 서빙** 을 살펴보겠습니다. 6.1.2에서 다룬 vLLM과 6.1.3에서 다룬 TGI는 양자화 모델을 직접 로드하여 서빙할 수 있습니다.

vLLM은 GPTQ와 AWQ 양자화 모델을 모두 지원합니다. 양자화 모델을 서빙하는 방법은 일반 모델과 거의 동일합니다.

```bash
# GPTQ 양자화 모델 서빙
vllm serve TheBloke/Llama-2-7B-Chat-GPTQ \
    --quantization gptq

# AWQ 양자화 모델 서빙
vllm serve TheBloke/Llama-2-7B-Chat-AWQ \
    --quantization awq
```

`--quantization` 옵션으로 양자화 방식을 지정합니다. vLLM은 양자화된 가중치를 로드한 뒤, 추론 시 GPU 커널에서 역양자화하여 연산을 수행합니다. 양자화 모델은 가중치 크기가 작으므로, 같은 GPU에서 더 많은 동시 요청을 처리하거나, 더 작은 GPU에서 모델을 실행할 수 있습니다.

TGI도 GPTQ와 AWQ를 지원합니다. Docker 컨테이너로 실행할 때 모델 이름만 양자화된 모델로 바꾸면 됩니다. TGI는 모델 설정 파일에서 양자화 방식을 자동으로 감지하므로, 별도의 양자화 옵션 없이도 동작합니다.

GGUF 포맷은 vLLM과 TGI에서 직접 지원하지 않습니다. GGUF는 llama.cpp 생태계에 속하므로, llama.cpp를 기반으로 하는 서빙 도구를 사용해야 합니다. llama.cpp 자체가 HTTP 서버 기능을 내장하고 있으며, Ollama도 REST API를 제공합니다. 다만 llama.cpp 기반 서버는 vLLM이나 TGI에 비해 대규모 동시 요청 처리 기능(연속 배칭, 고급 스케줄링 등)이 제한적입니다.

마지막으로 **실무 양자화 방식 선택 가이드** 를 정리하겠습니다. 양자화 방식의 선택은 크게 세 가지 기준에 따라 달라집니다. 실행 환경, 품질 요구 수준, 그리고 서빙 인프라입니다.

첫째, 실행 환경입니다. NVIDIA GPU가 있는 서버 환경이라면 GPTQ나 AWQ가 적합합니다. GPU의 병렬 연산 능력과 최적화된 CUDA 커널을 활용하여 빠른 추론 속도를 얻을 수 있습니다. GPU가 없거나, Apple Silicon Mac, 개인 노트북 등 다양한 하드웨어에서 실행해야 한다면 GGUF가 사실상 유일한 선택입니다.

둘째, 품질 요구 수준입니다. 양자화로 인한 품질 저하를 최소화해야 하는 프로덕션 환경이라면 AWQ를 우선 고려합니다. GPTQ도 품질이 높지만, AWQ가 동일 비트에서 약간 더 안정적인 결과를 보이는 경향이 있습니다. 개인 사용이나 실험 목적이라면 GGUF의 Q4\_K\_M이나 Q5\_K\_M이 품질과 크기 사이의 적절한 균형을 제공합니다.

셋째, 서빙 인프라입니다. vLLM이나 TGI를 사용하여 대규모 서빙을 하는 환경이라면, GPTQ 또는 AWQ 양자화 모델을 사용해야 합니다. 이 프레임워크들이 GGUF를 지원하지 않기 때문입니다. 반대로 단일 사용자가 로컬에서 모델을 실행하는 경우에는 Ollama와 GGUF 조합이 설치와 사용이 간편합니다.

```lua
실무 양자화 방식 선택 흐름

GPU 서버 환경인가?
  |
  +-- 예 --> vLLM/TGI로 서빙하는가?
  |            |
  |            +-- 예 --> AWQ 또는 GPTQ 선택
  |            |          (품질 우선이면 AWQ)
  |            |
  |            +-- 아니오 --> 용도에 따라 AWQ/GPTQ/GGUF 선택
  |
  +-- 아니오 --> CPU 또는 Apple Silicon인가?
                  |
                  +-- 예 --> GGUF 선택
                  |          (메모리 여유 있으면 Q5_K_M,
                  |           부족하면 Q4_K_M)
                  |
                  +-- 기타 --> GGUF 선택 (가장 넓은 호환성)
```

정리하면, GPTQ는 열 단위 오류 보상으로 높은 양자화 품질을 제공하는 GPU 중심의 PTQ 기법이고, AWQ는 활성화 크기를 기준으로 중요 가중치를 보호하여 빠르고 안정적인 양자화를 달성하는 기법입니다. GGUF는 양자화 알고리즘이 아닌 파일 포맷으로, llama.cpp 생태계에서 CPU를 포함한 다양한 하드웨어에서 모델을 실행할 수 있게 합니다. GPU 서빙 환경에서는 AWQ나 GPTQ를, CPU나 개인 장치에서는 GGUF를 선택하는 것이 일반적입니다.

다음 단원인 6.4.1에서는 모델 레지스트리와 버전 관리를 다룹니다. 양자화를 포함한 다양한 변형 모델을 체계적으로 관리하고 배포하는 방법을 살펴봅니다.

이 단원을 마치면 GPTQ, AWQ, GGUF의 특성을 비교하고, 실행 환경과 요구 사항에 맞는 양자화 방식을 선택할 수 있습니다.
